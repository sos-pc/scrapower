## 🔴 Bugs critiques trouvés

### B11 — `requeue_stale()` bypass `transition()` ✅ Corrigé

**Fichier** : `domain.py` L184-221

Remplacé le raw SQL par `transition(TIMEOUT)` qui gère `can_retry`, `retries++`, token cleanup. 2 tâches coincées en production nettoyées.

### B12 — Blob `ref_count` jamais décrémenté → GC inefficace

**Fichier** : `blob_store.py` L96 (dead code), `domain.py` L241-291

`delete_blob()` n'est jamais appelé. `cleanup_expired()` décrémente de 1, mais le ref_count initial est 2 (store + create). Résultat : `ref_count` ne descend jamais à 0. `run_gc()` ne supprime jamais rien. Fuite disque.

**Fix** : `task_manager.create()` ne devrait pas bumper ref_count (déjà à 1), ou `cleanup_expired()` doit décrémenter de 2.

## 🟡 Anomalies

### A1 — `transition()` `else` ne nettoie pas token

`transition()` branche `else` (L262-266) ne met pas `current_assignment_token = NULL`. Si on ajoute `ASSIGNED → QUEUED` sans cas spécifique, le token persiste.

### A2 — `transition()` skip token check si `None`

L233-235 : si `assignment_token=None`, la vérification est sautée. Aucun appelant n'exploite ça actuellement.

### A7 — Duplication worker ~150 lignes

`execute_wasm`, `execute_python`, `_heartbeat`, `run_worker`, upload+submit — identiques ou quasi-identiques dans les 3 workers. Tout fix doit être appliqué 3 fois.

### A8 — `ref_count` double-comptage

`store_blob` (ref=1) + `task_manager.create()` (+1) = ref=2. `cleanup_expired` décrémente de 1 → ref=1 → GC ne nettoie jamais.
